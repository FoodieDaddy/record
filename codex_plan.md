# Smart Record 五阶段改进实施计划

本文档基于 2026-06-07 操作录屏、当前 worktree 代码和 `AGENTS.md` 约束整理。目标是把「空间记录 -> 策略提示 -> 行为画像 -> 身份沉淀」的主链路压稳，优先修复数据完整性和体验断点，再做视觉与命名收敛。

## 总体原则

- 当前事实和收敛目标分开表达。Redis 优先是运行期新增功能的方向，但当前创建、加入、部分记分、归档仍会访问 MySQL，文档和提交说明不能写成 MySQL 零参与。
- 所有运行时文案必须避开审核高风险词，继续使用「空间、识别码、接入、封存、航程、脉冲、黑匣子、策略」等安全词。
- 2C2G 预算下禁止新增高频全房间轮询、全量 JSON 重复序列化、无差别广播和阻塞式批量计算。
- 前端继续维持黑底、蓝光、克制终端风格，但先保证可读性、流程清晰和数据可信。
- 每阶段完成后都要做至少一次小程序模拟器视觉验收；涉及后端 Java 改动后使用项目 verify 技能或等价流程验证编译、容器健康和 API 可访问。

## 阶段 1：数据闭环与构建一致性

### 目标

修复封存后历史样本可能丢失的问题，确认视频运行包和当前 worktree 一致。该阶段是 P0，必须先于 UI 大改。

### 背景问题

- 房主封存后，前端在 `miniprogram/pages/room/room.js` 中仍会额外调用 `DELETE /room/{roomId}/quit`。
- 后端 `RoomServiceImpl.quitRoom` 对已结束空间执行退出时，会删除对应 `room_member` 记录。
- `/score/trend`、`/score/yield-log`、镜像 stats、身份等级都依赖 `room_member.quit_time/final_score`。删除历史成员记录会导致身份沉淀和黑匣子回放不完整。
- 视频中的身份页文案和当前 worktree 不完全一致，存在运行包版本或开发者工具缓存差异。

### 实施步骤

1. 修复房主封存后的前端调用链。
   - 修改 `miniprogram/pages/room/room.js`。
   - 房主执行 `POST /score/room/{roomId}/settle` 成功后，不再调用 `DELETE /room/{roomId}/quit`。
   - 成功封存后仅做本地状态清理、断开 WS、展示结算弹层或跳转航程档案。
   - 普通成员主动断开仍保留 `DELETE /room/{roomId}/quit`。

2. 加固后端已结束空间退出语义。
   - 修改 `backend/src/main/java/com/smartrecord/service/impl/RoomServiceImpl.java`。
   - `room.status != 0` 时，禁止删除 `room_member` 历史记录。
   - 只清理 Redis `sr:user:rooms:{uid}` 中的进行中空间映射。
   - 返回成功即可，不广播 `MEMBER_LEAVE`，不修改 `quit_time/final_score`。

3. 补充历史样本回归测试。
   - 优先新增 service 层测试或集成测试。
   - 覆盖流程：创建空间 -> 成员记录脉冲 -> 房主封存 -> 调用已结束空间退出 -> 查询趋势、yield-log、历史房间。
   - 断言 `room_member` 仍保留该用户记录，`final_score` 和 `quit_time` 不为空。

4. 封存后缓存失效。
   - 在 `ScoreServiceImpl.doSettleRoom` 和本局录结算路径中，封存完成后清理或刷新相关缓存。
   - 目标缓存包括镜像 stats：`sr:mirror:stats:{uid}`，策略缓存：`sr:fortune:{uid}:{date}`，身份等级表或对应重算任务。
   - 身份等级当前已有异步重算，需确认所有成员都进入重算列表。

5. 构建一致性确认。
   - 重新编译小程序运行包，确认视频中出现的「黑匣子摘要 / 装备协议 / 进入黑匣子回放」是否来自当前代码、旧缓存或另一分支。
   - 清理微信开发者工具缓存后复测四个 Tab 首屏。
   - 记录实际运行包 commit、基础库版本和后端 profile。

### 验收标准

- 房主封存后不会删除任何已结算 `room_member` 记录。
- 封存后立刻进入身份页，样本数、近期脉冲、黑匣子回放能包含刚封存的空间。
- `/score/trend`、`/score/yield-log`、`/room/history` 返回一致。
- 视频运行包文案和当前 worktree 可解释，不再出现无法定位来源的界面。

### 风险与注意

- 不要把普通成员「未封存前退出」改成历史样本，它仍是主动断开当前空间。
- 不要为了修前端流程而跳过后端加固；后端必须防止旧客户端继续删历史。

## 阶段 2：空间页主流程重排

### 目标

让空间页从「危险操作和记录操作混在一起」收敛为清晰的记录终端：先选择成员、写入脉冲、看轨迹，再进行封存或断开。

### 背景问题

- 视频中 active 空间页中段，用户先看到「发送识别码 / 封存航程」，下方才是记录输入。
- 空空间仍可进入封存确认，最终只显示 toast「暂无可封存的航程数据」。
- 危险操作和主操作同屏距离过近，容易误触。

### 实施步骤

1. 重排 active 页面结构。
   - 修改 `miniprogram/pages/room/room.wxml` 和 `room.wxss`。
   - 推荐顺序：空间识别区 -> 舰员席位 -> 记录脉冲 -> 脉冲轨迹 -> 本局录入状态 -> 更多操作。
   - 将底部操作栏改为「发送识别码」和「更多」或下沉到页面末尾。
   - 「封存航程」放入危险区域或确认抽屉内，不和写入按钮相邻。

2. 增加封存可用状态。
   - 在 `room.js` 中新增 `sealMeta` 或 `canSealRoom`。
   - 自由流转模式可参考 `scoreRecords.length`、`pulseTraces.length`、`roomInsight.transferCount`。
   - 本局录入模式需识别已生效 round record，不能只看自由流转记录。
   - `canSealRoom=false` 时，主控看到「暂无轨迹，不能封存」或「解散空空间」。

3. 拆分「封存」和「解散空空间」。
   - 有记录数据：确认弹窗标题为「确认封存航程」，说明会写入黑匣子。
   - 无记录数据：确认弹窗标题为「解散空空间」，说明不会生成航程档案。
   - 两条路径调用不同 API 或在后端显式区分，避免用户以为无数据也能产出历史档案。

4. 优化记录输入。
   - 已选目标成员后，输入区保持固定高度，按钮文本绝对居中，loading 不抖动。
   - 未选成员时按钮 disabled，提示「请选择舰员席位」。
   - 如果空间只有自己，记录输入区不应出现可误解的数值输入，改成「等待舰员接入」。

5. 强化封存后去向。
   - 有数据封存成功后，优先展示结算弹层，并提供「查看航程档案」。
   - 无数据解散成功后回到 idle，并给出简短提示「空空间已关闭」。

### 验收标准

- 用户首屏能明确知道当前空间识别码、成员、记录入口。
- 空空间不会生成「航程档案」预期。
- 危险操作必须经过确认，且确认弹窗能清楚展示成员数、轨迹数、结果。
- 16 人成员网格在小屏不溢出，分数大数字不撑破席位。

### 风险与注意

- 不要在页面滚动中高频 `setData`。
- 不要把全量 `room` 或大数组在 WS 高频事件中反复写入。
- 自动封存逻辑仍需同时识别 `events` 和 `batches`。

## 阶段 3：策略页等待、结果态与缓存联动

### 目标

让策略页从「仪式感强但等待偏长」收敛为稳定可用的策略终端。用户点击后应快速得到状态建议，即使 LLM 慢也有本地策略兜底。

### 背景问题

- 前端等待 API 和固定日志动画都完成后才进入 result。
- 后端 LLM HTTP 和 CompletableFuture 超时均为 60 秒，移动端体感过长。
- 视频中策略生成阶段持续二十多秒，用户容易误认为页面卡住。
- 结果态中有时出现「待同步」状态和分享按钮并存，完成边界不清。

### 实施步骤

1. 后端软超时兜底。
   - 修改 `backend/src/main/java/com/smartrecord/service/impl/FortuneServiceImpl.java`。
   - 将同步请求等待控制在 6-8 秒内；超时立即返回 `fallbackFortune`。
   - 保留 LLM 日志记录，但不要阻塞用户请求。
   - 后续可设计异步补全缓存，但不能在本阶段宣称已实现。

2. 前端生成态最短和最长时长。
   - 修改 `miniprogram/pages/fortune/fortune.js`。
   - 动画最短展示 1.2-1.8 秒即可，不必固定完整播放所有日志。
   - 设置前端最长等待阈值，例如 8-10 秒；超时后显示「本地策略已启用」并允许重试。
   - API 先返回时允许提前结束动画，避免为动画而等待。

3. 结果态有效性校验。
   - 只有 `verdict`、`buffs`、`debuffs` 至少满足一组有效内容时，进入 result。
   - `sourceLabel` 使用「本地策略 / 模型策略」等中文主标签，英文只做弱装饰。
   - `solarTermLabel` 这类技术标签如果为 `LOW-NOISE`、`CRUISE`，前端需映射为中文节奏窗口。

4. 刷新和缓存语义。
   - `?force=true` 仍允许绕过 4 小时缓存。
   - 封存后阶段 1 清理用户策略缓存后，策略页应能反映最新样本。
   - 重新推演按钮需要提示刷新频率，不要让用户误以为每次都是完全新模型判断。

5. 文案安全复核。
   - 保留前端二次替换。
   - 后端敏感词过滤命中必须丢弃 LLM 输出并使用 fallback。
   - 分享海报也走同一套 sanitize。

### 验收标准

- 点击策略核心后，正常网络下 2 秒内有可感知进展，8 秒内一定有可读策略。
- LLM 慢、失败或返回敏感词时，用户看到本地策略，不看到错误堆栈。
- 分享按钮只在完整结果可用时出现。
- 策略输出保持状态管理、行动建议、风险控制，不做结果预测。

### 风险与注意

- 不要把异步 LLM 后补写成已完成能力，除非确实实现后台补全和缓存更新。
- 降低超时后要观察 `fortune_log.duration_ms/source/error_msg`，避免日志误导。

## 阶段 4：命名体系与视觉质量门禁

### 目标

统一产品语言，修复明显视觉破相和样式隐患，让四个主 Tab 成为同一个系统，而不是四套相似皮肤。

### 背景问题

- 视频中同一历史链路出现「黑匣子回放 / 任务档案 / 航程档案」多套命名。
- 录屏中回放页曾出现右侧半露、底部白块、尺寸调试浮层。
- 当前 WXSS 中存在缺分号样式，例如 `score-records.wxss` 和 `settle.wxss` 多处 `color: var(--color-cyan)` 后缺少分号。
- 部分暗色文字可读性不足，关键文案透明度偏低。

### 实施步骤

1. 建立命名词表。
   - 维护或扩展 `miniprogram/utils/terminology.js`。
   - 推荐统一：
     - Tab：空间、策略、镜像、身份。
     - 历史列表入口：黑匣子回放。
     - 历史列表页：黑匣子回放。
     - 单场详情页：航程档案。
     - 空间 ID：识别码。
     - 结束记录：封存航程。
   - 禁止同一入口同时使用「任务档案」「脉冲日志」「黑匣子回放」三个主名。

2. 同步页面标题和入口文案。
   - `room.json` 可继续「空间」或改「空间终端」。
   - `fortune.json` 保持「策略核心」或改「策略终端」，不要使用偏离主线的标题。
   - `mirror/index.json` 可保留「全息观测舱」，但页面内需明确镜像定位。
   - `profile.json` 保持「身份终端」。
   - `score-records.json` 根据词表统一为「黑匣子回放」或「脉冲日志」。
   - `settle.json` 统一为「航程档案」。

3. 修复样式语法和白底兜底。
   - 给所有 WXSS 缺分号处补齐分号。
   - 所有页面根容器和 `page` 必须有黑底兜底。
   - 抽屉、弹层、二级页转场不得露出白底。
   - 移除任何调试尺寸浮层或开发环境标记。

4. 可读性审计。
   - 关键正文透明度不得低于 0.40。
   - 装饰英文可低到 0.28，但不能承载关键含义。
   - 分享卡长文必须压缩成摘要，避免导出图片截断。

5. 动效静默和性能审计。
   - 核心页面根节点继续绑定 `reduce-motion`。
   - `app.wxss` 的 `.reduce-motion` 兜底需覆盖动画和 transition。
   - Canvas、timer、长链 `setTimeout` 在 `onHide/onUnload` 清理。
   - 禁止 `transition: all`。

### 验收标准

- 同一功能入口在 Tab、列表、详情、分享卡中命名一致。
- 真机和模拟器均不出现白底、半露抽屉、调试尺寸浮层。
- 分享卡保存后内容完整，不出现省略到无法理解的长句。
- 关闭动效后，策略、镜像、空间页没有持续动画。

### 风险与注意

- 不要为了统一命名把已有安全词换成审核高风险词。
- 不要让英文 kicker 成为唯一可理解信息，关键功能必须中文表达。

## 阶段 5：画像、身份与复盘联动

### 目标

让封存后的数据真正驱动策略、镜像和身份，形成「记录 -> 复盘 -> 画像 -> 身份」闭环。

### 背景问题

- 视频中身份页数据矩阵和黑匣子回放方向正确，但封存后的新样本是否立即参与策略/镜像/身份，取决于缓存刷新和历史数据完整性。
- 镜像页分享卡判读偏长，样本少时需要更明显的可信度提示。
- 回放页的「低样本参考」提示有价值，但位置偏后，用户可能先读结论再看到可信度。

### 实施步骤

1. 封存后联动刷新。
   - 阶段 1 的缓存失效完成后，前端在封存成功回到身份/镜像/策略页时触发轻量刷新。
   - 身份页 `onShow` 已并行加载用户、趋势、等级、MBTI，需确认不会被本地旧缓存覆盖。
   - 镜像 stats 如果命中旧 Redis 缓存，封存后应失效或重新计算。

2. 身份首页信息分层。
   - 首屏保留身份档案、数据矩阵、黑匣子摘要。
   - 声音、动效、触感等低频设置折叠或放二级设置，避免挤压核心身份信息。
   - 「权限档案」这类含义不清入口改为更明确的「身份档案」或「档案详情」。

3. 黑匣子回放列表增强。
   - 列表卡片展示：识别码、封存时间、参与人数、我的净脉冲、最高/最低成员。
   - 有低样本风险时在列表页顶部显示「样本较少，趋势仅供复盘参考」。
   - 单场详情继续展示封存序列、任务复盘、数值演化、关系网络、行为信号。

4. 镜像可信度表达。
   - 样本少于 3 场时，不输出强结论。
   - 分享卡摘要根据样本数分级：
     - 0-2 场：只展示协议状态和待采集提示。
     - 3-9 场：展示低样本参考和短判读。
     - 10 场以上：展示完整画像摘要。
   - 雷达图 locked 态要清楚，不用过暗文字表达关键状态。

5. 指标一致性。
   - 统一「样本数」来源，优先使用 `room_member.quit_time IS NOT NULL` 的已封存空间。
   - 身份页累计脉冲、策略用户标签、镜像 stats、yield-log 的样本口径需一致。
   - 如果某模块使用最近 20 场、另一个用最近 50 场，页面要以中文说明「近期」而不是暗示全量。

### 验收标准

- 封存一场后，身份样本数、黑匣子回放、趋势列表能立即看到新增样本。
- 镜像页低样本时不输出过强人格结论。
- 策略页刷新后可使用新样本计算用户标签。
- 用户从空间封存进入身份/镜像/策略，感知是一条连续链路，而不是四个孤立页面。

### 风险与注意

- 趋势和画像不能承诺结果，只能表达复盘参考、行为信号和状态管理。
- 样本少时所有画像输出都要降调，避免伪精确。

## 建议执行顺序

1. 阶段 1 必须先做，尤其是封存后历史成员记录保护。
2. 阶段 2 和阶段 3 可以并行，但都依赖阶段 1 的封存数据可信。
3. 阶段 4 应在 UI 大改前建立词表，也可穿插修样式语法。
4. 阶段 5 放在数据闭环稳定后做，否则画像和身份会被旧样本问题干扰。

## 最小验收清单

- 创建空间、接入空间、自由流转记录、封存、查看航程档案全链路通过。
- 空空间封存不会生成历史样本，不会误导用户。
- 封存后 `room_member` 历史记录保留，趋势/身份/镜像可读到样本。
- 策略页 LLM 慢时 8 秒内有本地策略可读。
- 四个 Tab 页面无白底、无调试浮层、无关键文字过暗、无分享卡文本截断。
- 后端 Java 编译通过，Redis/MySQL/Docker 健康，关键 API 可访问。
